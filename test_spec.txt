1. CONTEXTE DU PROJET 
 
Vous postulez au poste de Développeur Full-Stack Senior au sein de la société B’right Tunisie en 
phase de construction d'un assistant IA conversationnel dédié à la conformité réglementaire 
pharmaceutique (EMA, FDA, ICH). 
 
Le produit repose sur une architecture RAG-Centric (Retrieval-Augmented Generation) qui permet à 
des équipes réglementaires et qualité d'interroger en langage naturel une base documentaire 
constituée de textes normatifs externes et de documents internes, en obtenant des réponses fiables, 
sourcées et traçables. 
 
Ce test technique simule la construction d'une portion réaliste de ce système. Vous serez évalué sur 
votre capacité à : 
– Concevoir et implémenter un mini-pipeline RAG fonctionnel 
– Structurer une API backend robuste avec FastAPI 
– Implémenter une logique multi-tenant simple 
– Construire une interface de chat React minimaliste et fonctionnelle 
– Livrer un code propre, documenté et maintenable

3. DÉTAIL DES PARTIES 
 
PARTIE 1 — BACKEND API + PIPELINE RAG 
⏱ Durée estimée : 45 min – 1h 
  Difficulté : ★★★☆☆ 
📋 Tâches demandées : 
– Créer un endpoint POST /query (FastAPI) 
qui accepte { question: str, tenant_id: str } 
– Implémenter un mini-pipeline RAG avec 
LangChain : chunking → embedding → 
stockage vectoriel → retrieval → génération 
– Utiliser au minimum 3 documents PDF ou 
TXT simulant des textes réglementaires (ex : 
extraits ICH Q10, EU GMP) comme corpus 
– L'API doit retourner : la réponse générée, les 
sources utilisées (nom du document + 
extrait), et le tenant_id 
– Le LLM peut être OpenAI GPT-3.5/4, Mistral 
API, ou un modèle open-source local 
(Ollama accepté) 
📦 Livrables attendus : 
– Code source Python du backend 
– Endpoint /query fonctionnel et testable (curl 
ou Postman) 
– 3 documents de test inclus dans le dépôt 
– README indiquant les prérequis et la 
commande de lancement 
✅ Critères d'évaluation : 
– Maîtrise de LangChain (ou LlamaIndex) : chaîne RAG correctement orchestrée 
– Qualité de la stratégie de chunking (taille, overlap) 
– Pertinence des sources retournées dans la réponse 
– Gestion des erreurs (document non trouvé, LLM timeout, etc.) 
– Structure du code : separation of concerns, services, routeurs FastAPI

PARTIE 2 — ARCHITECTURE MULTI-TENANT & SÉCURITÉ 
⏱ Durée estimée : 30 min – 45 min 
  Difficulté : ★★★★☆ 
📋 Tâches demandées : 
– Implémenter une isolation simple par 
tenant_id : chaque tenant ne peut interroger 
QUE ses documents 
– Créer un endpoint POST /documents/ingest 
qui permet d'indexer un nouveau document 
en précisant le tenant_id 
– Ajouter un système d'authentification 
minimal : API Key dans le header (X-API-
Key), avec validation côté serveur 
– Loguer chaque requête /query dans une 
table SQL (SQLite accepté) : timestamp, 
tenant_id, question, nb_sources, durée_ms 
– Implémenter un endpoint GET 
/metrics/{tenant_id} retournant le nombre de 
requêtes et la durée moyenne 
📦 Livrables attendus : 
– Isolation multi-tenant fonctionnelle et 
démontrée (test avec 2 tenants distincts) 
– Endpoint /documents/ingest opérationnel 
– Authentification par API Key active 
– Table de logs SQL peuplée après chaque 
requête 
– Endpoint /metrics fonctionnel 
✅ Critères d'évaluation : 
– Isolation correcte : un tenant ne peut pas accéder aux documents d'un autre (test obligatoire) 
– Qualité de l'implémentation de la gestion des clés API 
– Pertinence et complétude du schéma SQL de métrologie 
– Clarté du code de middleware d'authentification 
 
PARTIE 3 — INTERFACE FRONTEND (REACT) 
⏱ Durée estimée : 30 min – 45 min 
  Difficulté : ★★☆☆☆ 
📋 Tâches demandées : 
– Créer une interface de chat simple en React 
(CRA ou Vite) 
– L'interface doit permettre de saisir une 
question et afficher la réponse de l'API 
– Afficher les sources retournées par l'API 
dans un panneau latéral ou un bloc dédié 
(nom du doc + extrait) 
– Gérer les états : chargement 
(loader/spinner), erreur API, réponse vide 
– Le tenant_id peut être hardcodé ou 
sélectionnable via un simple menu déroulant 
📦 Livrables attendus : 
– Application React fonctionnelle, connectée à 
l'API backend 
– Affichage de la réponse et des sources 
– Gestion des états de chargement et d'erreur 
– Code propre et composants bien découpés 
✅ Critères d'évaluation : 
– Fonctionnalité : la chaîne complète frontend → API → RAG fonctionne 
– UX minimaliste mais claire (pas besoin de design élaboré) 
– Gestion correcte des états asynchrones (async/await, loading, error) 
– Découpage en composants React pertinent

4. CONSIGNES DE LIVRAISON 
4.1 Format de rendu 
– Dépôt GitHub public (ou privé avec accès partagé) OU archive ZIP 
– Structure racine recommandée : /backend, /frontend, /docs 
– Un fichier README.md (le document peut être rédigé en français ou en anglais) à la racine 
expliquant comment lancer le projet 
– Variables d'environnement gérées via un fichier .env.example (ne pas committer la clé API réelle) 
4.2 Règles importantes 
– Vous pouvez utiliser toutes les bibliothèques open-source que vous jugez pertinentes 
– L'usage d'IA (Copilot, ChatGPT) est fortement  recommandée — mais vous devrez expliquer et 
défendre vos choix en entretien 
– En cas de blocage sur une partie, livrez ce que vous avez avec un commentaire expliquant 
l'obstacle rencontré — la démarche compte 
4.3 Critères de disqualification 
– Code entièrement copié-collé sans compréhension démontrée 
– Aucun fichier de documentation ou README 
– L'application ne démarre pas du tout

5. PROCESSUS D'ÉVALUATION 
 
Le test sera évalué en deux temps : 
 
Évaluation technique (J+2 à J+4 après la livraison) 
– Revue du code source : structure, lisibilité, robustesse 
– Exécution du projet selon les instructions du README 
– Vérification du fonctionnement de chaque partie 
– Revue de la grille de notation (barème ci-dessus) 
 
Entretien technique de restitution (60 min) 
– Présentation rapide de vos choix architecturaux 
– Questions sur des sections spécifiques du code 
– Discussion sur les limites de votre implémentation et les pistes d'amélioration 
– Question ouverte : « Comment rendriez-vous ce système conforme aux exigences de 
souveraineté des données (Cloud UE / HDS/ RGPD) ? » 
 
 
6. RESSOURCES UTILES 
 
Ces ressources vous sont fournies à titre indicatif — libre à vous d'utiliser d'autres sources.
